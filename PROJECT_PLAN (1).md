# CritCom — Technical Plan

CritCom is a FHIR-native agent that detects critical findings in radiology reports, dispatches notifications to the ordering provider, tracks acknowledgment, and escalates to on-call coverage when the ack window lapses.

It supports two ingestion paths — FHIR `DiagnosticReport` (primary) and DICOM Modality Worklist via Orthanc (secondary) — both normalized to a single `CritComStudy` model. The audit trail (`Communication`, `Task`) is always written to FHIR, regardless of trigger source.

## 3. SHARP context propagation

FHIR credentials travel in A2A message metadata; they never appear in the LLM prompt.

```
A2A message
  └── params.message.metadata
        └── "https://<workspace>.promptopinion.ai/schemas/a2a/v1/fhir-context"
              ├── fhirUrl:   "https://<vm>:8080/fhir"
              ├── fhirToken: "<bearer>"
              └── patientId: "patient-001"
```

Flow:
1. `shared/middleware.py` validates `X-API-Key` and bridges A2A `metadata` into `params.metadata`.
2. `shared/fhir_hook.py::extract_fhir_context` runs as `before_model_callback`, reads the SHARP extension key, and writes `fhir_url` / `fhir_token` / `patient_id` into ADK session state.
3. Tools read these from `tool_context.state` at call time.

The extension URI is constructed from `PO_PLATFORM_BASE_URL`. The hook is used unchanged from `po-adk-python`.

## 6. Normalized study model

Both `fetch_report_fhir` and `fetch_report_dicom` return the same shape:

```python
class CritComStudy(BaseModel):
    source: Literal["fhir", "dicom"]
    study_uid: str
    patient_id: str
    service_request_id: str | None      # FHIR path
    accession_number: str | None        # DICOM path
    modality: str
    report_text: str | None             # FHIR only; DICOM is metadata-only
    priority: Literal["stat", "urgent", "routine"]
    acr_category: Literal["cat1", "cat2", "cat3", "none"] | None
    timestamp: datetime
```

Priority mapping:
- FHIR: `ServiceRequest.priority` → `stat` / `urgent` / `routine`
- DICOM: Requested Procedure Priority (0040,1003) → STAT / HIGH / ROUTINE → `stat` / `urgent` / `routine`

Priority drives queue order only. The ACR category is determined separately — by the classifier when report text is available, or by an extension/tag when set upstream.

## 7. Tool contracts

All tools follow the ADK signature `(..., tool_context: ToolContext)` and read `fhir_url` / `fhir_token` / `patient_id` from `tool_context.state`.

| Tool | Input | Output | Side effect |
|---|---|---|---|
| `fetch_report_fhir` | `report_id: str` | `CritComStudy` | None |
| `fetch_report_dicom` | `study_uid: str` | `CritComStudy` | None |
| `resolve_provider` | `service_request_id: str` | `Practitioner` (id, name, contact methods) | None |
| `dispatch_communication` | `study: CritComStudy`, `provider_id: str`, `channel: phone\|pager\|email` | `communication_id: str` | Writes FHIR `Communication` |
| `track_acknowledgment` | `communication_id: str`, `acr_category` | `task_id: str`, `due: datetime` | Writes FHIR `Task` |
| `escalate` | `task_id: str` | new `task_id: str`, `oncall_provider_id: str` | Marks original `Task` failed; writes new `Communication` + `Task` to on-call |
| `query_audit` | `service_request_id: str \| patient_id: str` | list of `{Communication, Task}` | None |

## 8. ACR category → ack window

| Category | Meaning | Ack window |
|---|---|---|
| Cat1 — Immediate | Life-threatening | 60 minutes |
| Cat2 — Urgent | Significant, not immediately life-threatening | 24 hours |
| Cat3 — Routine | No special communication required | Standard workflow |
| None | Normal / no finding | No action |

`track_acknowledgment` sets `Task.restriction.period.end = now + window` based on category.

## 9. Workflow

```
1. Trigger
   ├── FHIR: new DiagnosticReport with status="final"
   └── DICOM: Orthanc MWL shows study with completion status

2. fetch_report_{fhir,dicom}
   └── Read priority → enqueue → normalize to CritComStudy

3. (FHIR path only) classify
   └── Map report text → cat1/cat2/cat3/none

4. resolve_provider
   └── ServiceRequest.requester → Practitioner → contact methods

5. dispatch_communication
   └── Notify via channel; write FHIR Communication

6. track_acknowledgment
   └── Write FHIR Task with deadline per ACR category

7. Wait for ack
   ├── Acknowledged before deadline → mark Task complete
   └── Deadline passed → escalate

8. escalate
   └── Mark original Task failed
   └── Resolve on-call PractitionerRole
   └── dispatch_communication to on-call
   └── Create new Task; loop to step 7

9. query_audit (any time)
   └── Return Communication + Task history for a ServiceRequest or Patient
```

## 10. Agent instruction

```
You are CritCom, a critical results communication agent for radiology.

When asked to process a study:
1. Call fetch_report_fhir if the trigger is a FHIR DiagnosticReport ID, or
   fetch_report_dicom if the trigger is a DICOM study UID. Both return the
   same study object.
2. Call resolve_provider to find the ordering physician's contact details.
3. Call dispatch_communication to notify the physician and record the
   Communication in FHIR.
4. Call track_acknowledgment to create a Task and monitor for acknowledgment.
5. If the Task becomes overdue, call escalate to notify the on-call provider.
6. Call query_audit at any time to return the full communication history.

Always be precise about FHIR resource IDs. Always confirm when a Communication
has been dispatched and when a Task has been created. If a tool returns an
error, report it clearly and do not proceed to the next step.
```
